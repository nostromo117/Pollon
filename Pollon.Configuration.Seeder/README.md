# Pollon Configuration Seeder

This project is a small utility responsible for seeding the **Consul Key-Value store** with dynamic infrastructure configuration generated by .NET Aspire.

## Why this exists
In a distributed environment, services need to know where infrastructure like RabbitMQ or Redis is located. While Aspire manages these via dynamic environment variables, external plugins or independent microservices may prefer a centralized configuration store like Consul.

This seeder bridges that gap by reading Aspire's connection strings at startup and pushing them into Consul.

## Technical Details: The CONSUL_URL Gotcha

The project uses a custom environment variable **`CONSUL_URL`** to define the Consul endpoint (e.g., `http://localhost:8500`).

### Why not `CONSUL_HTTP_ADDR`?
The official `Consul` .NET client library has a built-in feature that automatically reads `CONSUL_HTTP_ADDR`. However:
1.  It uses a rigid internal parser that expects a `host:port` string (e.g., `127.0.0.0:8500`).
2.  When it receives a full URI containing a scheme (like the `http://...` provided by Aspire), the internal parser fails to extract the port correctly and throws a `FormatException`.

By using `CONSUL_URL`, we:
- Avoid triggering the library’s fragile automatic parsing.
- Use the robust .NET `Uri` class to handle the endpoint, ensuring stability regardless of the protocol or address format.

## Seeding Structure
The seeder currently populates:
- `pollon/messaging`: The RabbitMQ connection string.

## Configuration Provider
To consume these values, use the `AddConsulConfiguration()` extension method available in `Pollon.ServiceDefaults`.
